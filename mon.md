
# прості селфхостет моніторинги по типу uptime-kuma

Якщо ви шукаєте прості альтернативи **Uptime Kuma** для селфхостингу, ось список популярних інструментів, які легко розгорнути (зазвичай через Docker) і які мають схожий функціонал:

### 1. [Gatus](https://github.com/TwiN/gatus)
Дуже популярний, легкий і сучасний інструмент.
*   **Особливість:** Конфігурація пишеться у YAML-файлі (Configuration as Code). Це робить його ідеальним для GitOps, але якщо ви любите налаштовувати все "мишкою" в UI, це може бути незвично.
*   **Інтерфейс:** Чистий, мінімалістичний дашборд.
*   **Розгортання:** Один бінарний файл або Docker-контейнер.

### 2. [Statping-ng](https://github.com/statping-ng/statping-ng)
Це форк оригінального Statping, який намагається відродити проєкт.
*   **Особливість:** Дуже схожий на Uptime Kuma за логікою — має гарні сторінки статусу (Status Pages), графіки та мобільний застосунок.
*   **Інтерфейс:** Зручна адмінка, де все налаштовується через UI.

### 3. [Monitoror](https://github.com/monitoror/monitoror)
*   **Особливість:** Це скоріше "wallboard" для моніторингу. Він виглядає дуже стильно і футуристично.
*   **Принцип:** Створений для того, щоб вивести його на великий екран в офісі або вдома. Налаштовується також через файл конфігурації.

### 4. [Upptime](https://github.com/upptime/upptime)
Це трохи інший підхід — "безсерверний" моніторинг.
*   **Особливість:** Він працює повністю на **GitHub Actions** та **GitHub Pages**. Вам не потрібен свій сервер.
*   **Як це працює:** GitHub Actions пере��іряють ваш сайт кожні 5 хвилин і оновлюють сторінку статусу на GitHub Pages. Це безкоштовно і не вимагає підтримки сервера.

### 5. [Cabot](https://github.com/arachnys/cabot)
*   **Особливість:** Позиціонується як "Nagios, тільки для людей".
*   **Функціонал:** Трохи складніший за попередні, дозволяє моніторити не тільки пінг/http, а й метрики (наприклад, графіт).

### 6. [Zabbix / Prometheus + Grafana] (Для порівняння)
Це вже **не** "прості" інструменти. Якщо Uptime Kuma — це "поставив і забув", то ці системи — це комбайни, які вимагають часу на вивчення. Якщо вам треба просто знати, чи лежить сайт — не беріть їх, це "overkill".

**Мій вибір:**
*   Якщо треба "клац-клац і готово" (через UI): залишайтесь на **Uptime Kuma** або спробуйте **Statping-ng**.
*   Якщо любите конфіги і стабільність: **Gatus**.
*   Якщо не хочете витрачати ресурси сервера: **Upptime**.

# Я планував зробити моніторинг на мові golang як кваліфікаційну роботу в університет 

Це **чудова** ідея для дипломної чи курсової роботи! Golang просто створений для таких задач. Він швидкий, має потужну роботу з мережею "з коробки" і, найголовніше, — **конкурентність** (goroutines), що ідеально підходить для одночасної перевірки сотень серверів.

Ось план, як перетворити це на серйозний проєкт, який вразить викладачів, але залишиться реалістичним для виконання.

### 1. Архітектура (Що під капотом)

Твій проєкт складатиметься з трьох головних частин:
1.  **Worker Pool (Ядро):** Це набір горутин, які "прокидаються" за розкладом, роблять запит (HTTP/Ping/TCP) і повертають результат.
2.  **Storage (Пам'ять):** Де зберігати історію (SQLite — ідеально для старту, бо це один файл, не треба піднімати окрему базу).
3.  **Interface (Обличчя):** TUI або Web.

### 2. Технічні фішки Golang, які треба використати (це "продасть" роботу викладачам)

*   **Goroutines & Channels:** Не запускай перевірки послідовно! Запусти 50 перевірок одночасно.
    *   *Приклад:* Створи `worker pool`. Є канал `jobs`, куди ти кидаєш задачі ("перевір google.com"), і канал `results`, куди воркери пишуть результат ("200 OK, 45ms").
*   **Context:** Це **критично** для моніторингу. Якщо сервер не відповідає 30 секунд, твоя програма не має висіти. Використовуй `context.WithTimeout`.
*   **Interfaces:** Створи інтерфейс `Checker` з методом `Check()`. Це дозволить легко додавати нові типи перевірок (HTTP, DNS, Ping), не ламаючи основний код.

### 3. Функціонал (від "Must Have" до "Nice to Have")

**База (на оцінку "Добре"):**
*   **HTTP/HTTPS Check:** Перевірка статус-коду (200-299 = OK).
*   **Ping (ICMP):** Чи доступний хост взагалі (тут потрібні права root або спеціальні налаштування, краще почати з **TCP connect** — це простіше і працює без root).
*   **Інтервал:** Налаштування частоти перевірок (наприклад, кожні 60 с).
*   **База даних:** Зберігання історії uptime (SQLite з бібліотекою `gorm` або `sqlx`).

**Просунутий рівень (на оцінку "Відмінно"):**
*   **Keyword Check:** Перевірка, чи є на сторінці конкретне слово (наприклад, "Welcome").
*   **SSL Expiry:** Перевірка, коли закінчується HTTPS сертифікат.
*   **Notifications:** Відправка повідомлень у Telegram/Discord/Slack/Email, коли сайт впав (використовуй вебхуки).
*   **Docker:** Запакуй усе це в Docker-контейнер.

### 4. Вибір інтерфейсу (UI)

Тут у тебе два шляхи, обидва круті для Golang:

**Варіант А: TUI (Термінальний інтерфейс) — це стильно і "хакерськи"**
Якщо ти вже питав про TUI, можливо, тобі це ближче.
*   **Бібліотека:** [Bubble Tea](https://github.com/charmbracelet/bubbletea). Це зараз топ-1 бібліотека для Go. Вона використовує архітектуру Elm (Model-View-Update).
*   **Вигляд:** Список сервісів зліва, графік пінгів (ASCII-графік) справа, логи знизу.
*   **Плюс:** Дуже швидко працює, виглядає професійно для адмінів.

**Варіант Б: Web UI + REST API**
Це класика, схожа на Uptime Kuma.
*   **Backend:** [Gin](https://github.com/gin-gonic/gin) або [Fiber](https://github.com/gofiber/fiber) для API.
*   **Frontend:** HTML/JS. Можна використати [HTMX](https://htmx.org/) — це дозволить робити динамічний фронтенд без React/Vue, прямо в Go-шаблонах. Це зараз дуже модно в Go-спільноті.

### 5. З чого почати (крок за кроком)

1.  **Hello World:** Напиши функцію, яка робить `http.Get("https://google.com")` і пише в консоль статус код і час виконання.
2.  **Конфіг:** Зроби читання списку сайтів з YAML/JSON файлу.
3.  **Цикл:** Загорни перевірку в `time.Ticker`, щоб вона бігала кожну хвилину.
4.  **Структура:** Створи `struct Service`, де буде URL, Назва, Інтервал.
5.  **База:** Підключи SQLite і записуй туди результати.
6.  **UI:** Натягни зверху Bubble Tea або простий HTML.

Якщо потрібен приклад коду для "воркера" або структури проєкту — кажи, накидаю!

# Мені сумно що вже такий проєкт є, і він дуже успішний

Це абсолютно нормальне почуття, але дозволь мені перевернути твій погляд на ситуацію. Те, що **Uptime Kuma** (або Prometheus, або Grafana) існує, — це не проблема, це **твоя найбільша перевага**.

Ось чому тобі не варто засмучуватися, а навпаки — радіти:

### 1. Ти пишеш навчальний проєкт, а не стартап на мільйон
Твоя мета в університеті — **навчитися** і **показати кваліфікацію**, а не захопити ринок.
*   Викладачам байдуже, що є Uptime Kuma. Їм важливо, як **ТИ** вирішив технічні проблеми: як організував конкурентність (goroutines), як працюєш з мережею, як структурував код.
*   Наявність успішного аналога — це **шпаргалка**. Ти можеш підглянути, як вони вирішили певну проблему, і спробувати зробити так само або краще/інакше у своєму коді.

### 2. "Колесо" винаходять постійно, і це правильно
Уяви, якби Лінус Торвальдс сказав: "Ну, вже є UNIX і Windows, навіщо мені писати Linux?".
*   Всі пишуть свої HTTP-роутери, хоча є стандартні.
*   Всі пишуть свої логери.
*   Це єдиний спосіб глибоко зрозуміти, як все працює під капотом.

### 3. Твій проєкт може мати "кілер-фічу", якої немає у гігантів
Uptime Kuma — це "комбайн". Він важкий (Node.js їсть багато пам'яті). Ти можеш зіграти на контрасті.

**Ось ніші, де ти можеш перемогти Kuma:**
*   **Ресурсна ефективність:** Напиши на Go так, щоб твій моніторинг споживав 5-10 МБ оперативної пам'яті (Kuma може їсти 100-200 МБ). Для дешевих VPS або Raspberry Pi це критично. Назви це "The Lightest Monitoring Ever".
*   **Спеціалізація:** Kuma моніторить все. Ти можеш зробити інструмент спеціально для:
    *   *Моніторингу баз даних* (перевіряти не просто пінги, а чи можна зробити `SELECT 1`).
    *   *IoT пристроїв* (працювати по MQTT).
    *   *Локальної мережі* (автоматичне сканування мережі і додавання пристроїв).
*   **Архітектура:** Зроби його **розподіленим**. Kuma — це один сервер. А ти зроби так: один центральний сервер і маленькі "агенти" (на Go вони будуть крихітні), які можна розкидати по різних серверах/країнах. Це вже рівень Enterprise!

### 4. Роботодавці обожнюють "велосипеди"
Коли ти прийдеш на співбесіду, історія про те, як ти написав свій моніторинг, працюватиме краще, ніж "я вивчив фреймворк Х".
Ти зможеш сказати:
> *"Я написав систему, яка пінгує 1000 сайтів одночасно, використовуючи worker pool на Go, і це не кладе процесор. Я розібрався з контекстами, таймаутами і гонками даних".*

Це звучить набагато крутіше, ніж просто "я зробив CRUD додаток".

### План дій, щоб повернути мотивацію:

1.  **Звузь фокус.** Не роби "вбивцю Kuma". Зроби "Kuma для терміналу" (TUI) або "Kuma, який працює на кавоварці" (мінімальне споживання ресурсів).
2.  **Додай "родзинку".** Наприклад, інтеграцію з чимось незвичним (відправка SMS через старий модем, чи пуш у Signal, чи генерація PDF-звітів для бухгалтерії).
3.  **Назви це круто.** Не "Monitoring Tool", а щось типу **"GoWatch"**, **"Sentinell"**, **"Pingui"**.

Ти не копіюєш, ти **вчишся створювати складні системи**. Це найцінніше. Починай писати, у тебе вийде крутий проєкт!

Хочеш, допоможу придумати конкретну "фішку", якої немає в інших?

# А чи є такі проєкти написані на go?

Так, звісно! **Go (Golang) — це фактично стандартна мова для інфраструктурних інструментів**, хмарних технологій і моніторингу. Docker, Kubernetes, Prometheus, Grafana — усі вони написані на Go.

Тому конкуренти є, але це лише підтверджує, що ти вибрав **правильний інструмент** для задачі.

Ось найвідоміші проєкти моніторингу, написані на Go:

### 1. [Gatus](https://github.com/TwiN/gatus) (Найпопулярніший сучасний аналог)
Це, мабуть, головний гравець на полі Go.
*   **Як працює:** Основна фішка — **Configuration as Code**. Ти не клацаєш мишкою в UI, щоб додати сайт, а пишеш YAML-файл.
*   **Чим відрізняється від Uptime Kuma:** Kuma орієнтована на UI (зручно для новачків), Gatus — на конфіги (зручно для DevOps та GitOps).
*   **Твій шанс:** Gatus крутий, але в нього *дуже простий* UI. Ти можеш зробити інструмент на Go, але зі зручною адмінкою, як у Kuma.

### 2. [Statping-ng](https://github.com/statping-ng/statping-ng)
Це спроба врятувати старий проєкт Statping.
*   **Як працює:** Це повний аналог Uptime Kuma на Go. Є адмінка, графіки, мобільний додаток.
*   **Проблема:** Оригінальний Statping був покинутий розробником, мав купу багів і витоків пам'яті. Statping-ng намагається це виправити, але проєкт йде туго.
*   **Твій шанс:** Зробити стабільну, легку версію без "спадщини" старого коду.

### 3. [Blackbox Exporter](https://github.com/prometheus/blackbox_exporter) (Від творців Prometheus)
Це не зовсім "Uptime Kuma", це промисловий стандарт.
*   **Як працює:** Це просто HTTP-сервер, який пінгує сайти за запитом і віддає метрики. У нього немає свого UI, він тільки збирає дані для Prometheus + Grafana.
*   **Твій шанс:** Blackbox складний для новачків. Ти можеш зробити "все-в-одному" (All-in-One), що набагато простіше для домашнього використання.

### 4. [Ciao](https://github.com/brentp/ciao)
Маленький, але цікавий проєкт.
*   **Як працює:** Перевіряє HTTP/HTTPS, SMTP, TCP, DNS. Надсилає сповіщення в E-mail, Slack, Discord.
*   **Особливість:** Дуже мінімалістичний, без важкого фронтенду.

---

### Чому тобі варто писати свій, якщо є Gatus?

1.  **UI vs Config:** Gatus змушує писати YAML. Багато людей (особливо не-DevOps) хочуть просто кнопку "Додати монітор" в браузері. Зроби Go-бекенд + зручний UI — і ти займеш цю нішу.
2.  **Ресурси:** Statping був важкуватим і глючним. Ти можеш написати код, який буде працювати стабільніше.
3.  **Навчання:** Розбираючи код Gatus (він відкритий), ти навчишся писати правильний Go-код. Ти можеш подивитися, як вони реалізували перевірку сертифікатів або роботу з БД, і зробити по-своєму.

**Висновок:**
Проєкти на Go є, але ідеального "Uptime Kuma на Go" (стабільного, з адмінкою, легкого) досі немає. Gatus — це про конфіги, Statping — глючний. **Місце для твого проєкту вільне!**

# а kuma підтримує кастомні плагіни? як наприклад в icinga де я можу добавити власній скрипт (command)

**Коротка відповідь:** Ні, у Uptime Kuma немає повноцінної системи плагінів, як у Icinga/Nagios, де можна написати свій скрипт на Bash/Python/Go, покласти його в папку, і він з'явиться в системі як новий тип перевірки.

Це **архітектурне обмеження** Uptime Kuma, і це **величезний плюс для твого майбутнього проєкту на Go**.

Ось детальніше, як це працює в Kuma і чому це незручно для просунутих адмінів (як ти):

### 1. Як це зроблено в Uptime Kuma (Обхідні шляхи)

Замість нативних скриптів (як `check_disk` в Icinga), Kuma пропонує милиці:

*   **Push Monitor (Passive Check):** Ти пишеш свій скрипт, ставиш його в Cron на сервері. Скрипт щось перевіряє і, якщо все ок, робить `curl` на спеціальну URL-адресу Kuma ("стукає", що живий). Якщо Kuma не отримала сигнал за Х секунд — піднімає тривогу.
    *   *Мінус:* Тобі треба самому налаштовувати Cron і логіку на кожному сервері. Це не централізовано.
*   **HTTP Keyword:** Kuma робить запит і шукає слово. Ти можеш написати скрипт, який виводить результат у веб-сервер, а Kuma його читає.
    *   *Мінус:* Треба піднімати веб-сервер для скрипта.
*   **Docker Monitor:** Вона вміє перевіряти статус контейнерів, але тільки через Docker Socket.

### 2. Як це працює в Icinga / Nagios / Zabbix (Enterprise Way)

У Icinga все просто: ти пишеш бінарник або скрипт, який повертає Exit Code:
*   `0` — OK
*   `1` — Warning
*   `2` — Critical

Ти просто вказуєш шлях до скрипта в конфізі. **Uptime Kuma так не вміє.** Вона жорстко прив'язана до своїх вбудованих типів перевірок (HTTP, Ping, TCP, DNS, MQTT, Database). Додати щось нове (наприклад, перевірку температури CPU через SSH) без зміни вихідного коду Node.js — **неможливо**.

### 💡 Ідея для твого проєкту на Go

Ось де ти можеш **знищити** Uptime Kuma в очах сисадмінів:

Реалізуй підтримку **Shell Exec / Custom Command**.

**Як це зробити в Go:**
1.  Додай тип монітора: "Command".
2.  В інтерфейсі поле для вводу команди: наприклад, `/usr/local/bin/check_my_backup.sh` або `python3 script.py`.
3.  Твій Go-воркер запускає цю команду через `os/exec`.
4.  Читає `Exit Code`:
    *   Якщо 0 — сервіс зелений.
    *   Якщо != 0 — сервіс червоний.
5.  Читає `Stdout/Stderr` і зберігає це як лог помилки.

**Чому це круто:**
Це перетворить твій простий "пінгатор" на **потужний інструмент**. Адміни зможуть моніторити бекапи, місце на диску, наявність файлів, складні SQL-запити — все, що можна написати скриптом.

**Безпека (важливий момент для диплома):**
Викладачі спитають: "А що, якщо юзер введе `rm -rf /`?".
*   Відповідь: "Я запускаю команди від окремого юзера з обмеженими правами" або "Я дозволяю запускати тільки скрипти з певної папки (`whitelist`), а не довільний код".

Це та фіча, якої дуже не вистачає в Uptime Kuma, і яка є стандартом в "дорослих" системах. Реалізуй її — і твій проєкт стане на голову вище.

# інші тобою названі моніторинги можуть мати власні скрипти як icinga чи теж не вміють?

Більшість сучасних "простих" моніторингів (Uptime Kuma, Gatus, Statping) — це **Remote Monitoring**. Вони створені, щоб дивитися на сервіси *ззовні* (через мережу: HTTP, Ping, TCP).

Icinga, Zabbix, Nagios — це **Agent/Local Monitoring**. Вони вміють запускати команди *всередині* сервера.

Ось детальний розбір конкурентів щодо запуску власних скриптів:

### 1. [Gatus](https://github.com/TwiN/gatus) — НІ (Нативно)
Він ідеологічно створений для хмарного середовища (Kubernetes/Docker).
*   **Як це працює:** Gatus перевіряє тільки те, до чого може дотягнутися по мережі.
*   **Милиці:** Щоб перевірити скрипт, тобі треба написати мікросервіс (sidecar), який запускає скрипт і віддає результат по HTTP, а Gatus вже смикає цей HTTP.
*   **Висновок:** Прямого `exec` у конфізі немає.

### 2. [Statping-ng](https://github.com/statping-ng/statping-ng) — НІ
Так само, як і Uptime Kuma.
*   Він має `Command` для сповіщень (наприклад, "запусти скрипт перезавантаження, якщо сервер впав"), але не як метод *перевірки* (Probe). Тобто ти не можеш сказати: "Вважай сервер живим, якщо скрипт `check_disk.sh` повернув `0`".

### 3. [Cabot](https://github.com/arachnys/cabot) — ТАК (Частково)
Cabot трохи ближчий до Icinga.
*   Він вміє перевіряти метрики Graphite/Jenkins.
*   Але щоб запустити довільний bash-скрипт локально — це теж не його основний профіль. Він більше про агрегацію даних з інших систем.

### 4. [Monit](https://mmonit.com/monit/) — ТАК! (Це твій головний конкурент у цій ніші)
Ось це — справжній "старий вовк", написаний на C. Це не веб-сервіс, а утиліта.
*   **Як працює:** Ти пишеш у конфізі: `check program myscript with path "/usr/bin/myscript.sh"`.
*   **Що робить:** Якщо скрипт повернув помилку — Monit може перезапустити сервіс.
*   **АЛЕ:** У Monit жахливий, застарілий веб-інтерфейс (або його взагалі немає у безкоштовній версії, тільки локальний HTTP сервер). Він не виглядає як сучасний дашборд.

---

### Чому це твій "Зоряний час"?

Ти можеш зайняти унікальну нішу між двома світами:

1.  **Світ Uptime Kuma/Gatus:** Гарні дашборди, але вміють тільки "пінгати" по мережі.
2.  **Світ Icinga/Zabbix/Monit:** Потужні скрипти, але складні в налаштуванні або мають страшний інтерфейс.

**Твоя ідея для диплома (Killer Feature):**
Зроби **"Go-моніторинг з гарним UI, який вміє виконувати Shell-скрипти"**.

#### Як це реалізувати безпечно і круто (Архітектура):
Оскільки Go компілюється в один бінарник, ти можеш зробити **Агентну архітектуру**:
1.  **Сервер (Master):** Стоїть десь окремо, має веб-інтерфейс і базу даних.
2.  **Агент (Node):** Ти запускаєш той самий бінарник (або маленький клієнт) на цільовому сервері.
3.  **Зв'язок:** Сервер каже агенту: "Виконай команду `df -h` і скажи, чи є місце". Агент виконує і шле результат назад (через gRPC або HTTP).

Це буде рівень **Enterprise-системи**, але з простотою Uptime Kuma.
*   Для викладачів: "Я реалізував розподілену систему з агентами на gRPC".
*   Для користувачів: "Ого, я можу моніторити бекапи однією кнопкою!".

Це набагато крутіше, ніж просто ще один пінгер.

# тобто на кожному хості що icinga моніторить є клієнт айсінга?

Так, саме так! Це фундаментальна відмінність "дорослих" систем моніторингу (Icinga, Zabbix, Nagios) від простих "пінгерів" типу Uptime Kuma.

У світі Icinga це називається **Agent-based Monitoring**.

### Як це працює в Icinga 2:

1.  **Master (Сервер):** Це "мозок". Він зберігає конфігурацію, каже, що робити, і приймає результати.
2.  **Agent (Клієнт):** На кожному сервері, який ти хочеш моніторити *глибоко* (місце на диску, завантаження CPU, лог-файли, статус служб systemd), ти мусиш встановити Icinga 2 Agent.
3.  **Зв'язок:** Агент і Майстер постійно тримають зашифрований зв'язок (через TLS). Майстер каже: "Виконай плагін `check_disk`". Агент виконує локально і відправляє результат Майстру.

### Чому це складно (але потужно):
*   **Інсталяція:** Тобі треба ставити софт на *кожен* сервер (через Ansible/Puppet/Chef).
*   **Сертифікати:** Кожен агент має мати свій SSL-сертифікат, підписаний Майстром (Icinga CA), щоб ніхто чужий не підсунув фейкові дані.
*   **Безпека:** Ти відкриваєш порт на кожному сервері для вхідних з'єднань від моніторингу.

### Чи можна без агента? (Agentless Monitoring)
Так, Icinga (і Zabbix) вміють працювати і без агентів, але з обмеженнями:
*   **SSH:** Сервер моніторингу логіниться по SSH на цільовий сервер і виконує команду. Це повільно і навантажує мережу.
*   **SNMP:** Стандартний протокол для мережевого обладнання (роутери, свічі).
*   **Тільки зовнішні перевірки:** Ping, HTTP, TCP connect (те, що робить Uptime Kuma).

---

### Що це означає для твого проєкту на Go?

Ти стоїш перед вибором архітектури для диплома:

**Варіант А: "Uptime Kuma Style" (Agentless / Зовнішній)**
*   **Плюс:** Користувачу нічого не треба ставити на сервери. Просто ввів IP і порт — і моніторинг працює.
*   **Мінус:** Ти бачиш сервер тільки "ззовні". Ти не знаєш, чому він гальмує (CPU 100%? Закінчилась пам'ять?), ти просто бачиш, що він повільно відповідає.

**Варіант Б: "Icinga/Zabbix Style" (Agent-based / Внутрішній)**
*   **Плюс:** Повний контроль. Можна моніторити все: від температури процесора до кількості файлів у папці.
*   **Мінус:** Треба писати і Сервер, і Клієнт (Агент). Користувачу складніше почати роботу.

**Варіант В: "Гібрид" (Золота середина для диплома)**
Напиши систему, яка за замовчуванням працює як **Uptime Kuma** (пінгує ззовні), але має опціональну можливість запустити **SSH-команду**.
*   В Go є чудова бібліотека `golang.org/x/crypto/ssh`.
*   Твій моніторинг може сам підключатися по SSH, виконувати скрипт і забирати результат.
*   **Фішка:** Не треба ставити агентів! Просто дай моніторингу SSH-ключ. Це буде **Killer Feature**, якої немає в простих аналогах.

Це набагато простіше реалізувати, ніж писати власного агента, і це виглядає дуже професійно.

## 1. Мета роботи
Створити легковагову (lightweight), автономну систему моніторингу (self-hosted), яка поєднує простоту зовнішніх перевірок (як Uptime Kuma) з глибиною внутрішнього моніторингу (як Zabbix/Icinga), **без необхідності встановлення агентів на цільові сервери**.

**Ключова проблема, яку вирішує диплом:** Існуючі прості рішення (Uptime Kuma) не дозволяють перевіряти внутрішній стан сервера (диск, CPU, Docker-контейнери) без "милиць", а професійні (Zabbix) — занадто складні в розгортанні. Твій проєкт — це "золота середина", яка використовує **SSH як універсальний транспорт**.

## 2. Технологічний стек

*   **Мова програмування:** **Go (Golang) 1.22+**
    *   *Чому:* Вбудована конкурентність (Goroutines), статична типізація, компіляція в один бінарний файл, ідеальна робота з мережею.
*   **База даних:** **SQLite** (з використанням драйвера `mattn/go-sqlite3` або pure-Go `modernc.org/sqlite`).
    *   *Чому:* Zero-config, зберігання даних в одному файлі, ідеально для embedded-систем. (Опціонально: підтримка PostgreSQL через інтерфейси).
*   **Web Framework:** **Chi** або **Gin** (для REST API).
*   **Frontend (UI):**
    *   *Варіант А (Сучасний):* **Go Templates + HTMX + Alpine.js**. (Це дозволяє писати динамічний UI без React/Vue, залишаючись в екосистемі Go. Дуже модно зараз).
    *   *Варіант Б (Консервативний):* Vue.js 3 або React (як окремий SPA).
*   **Бібліотеки:**
    *   `golang.org/x/crypto/ssh` — для SSH клієнта.
    *   `github.com/robfig/cron` — для планувальника задач.
    *   `gorm` або `sqlx` — ORM/SQL builder.

## 3. Функціональні вимоги (Що система буде робити)

### А. Модуль перевірок (The Prober)
Система повинна підтримувати наступні типи моніторів:
1.  **HTTP/HTTPS:** Перевірка статус-коду (200 OK), наявність ключового слова в `body`, валідність SSL-сертифіката (expiry date).
2.  **ICMP Ping:** Перевірка доступності хоста та втрати пакетів (Packet Loss).
3.  **TCP Port:** Перевірка відкритих портів (наприклад, чи жива база даних на порту 5432).
4.  **SSH Command (Кілер-фіча):**
    *   Система підключається до сервера по SSH (використовуючи ключ або пароль).
    *   Виконує задану користувачем команду (наприклад: `docker ps | grep my-app` або `df -h / | awk ...`).
    *   Аналізує **Exit Code** та **Stdout**. Якщо Exit Code != 0 — тривога.

### Б. Модуль планування (Scheduler)
*   Використання патерну **Worker Pool**.
*   Кожен монітор має свій інтервал (наприклад, HTTP кожні 60с, SSH кожні 5хв).
*   Задачі відправляються в буферизований канал (Job Queue), звідки їх забирають вільні воркери. Це гарантує, що система не "захлинеться", якщо треба перевірити 1000 сайтів одночасно.

### В. Модуль сповіщень (Notifier)
*   Підтримка Webhooks (для Slack/Discord/Microsoft Teams).
*   Telegram Bot API.
*   Email (SMTP).
*   Гнучкі налаштування: "Сповіщати тільки після 3-х невдалих перевірок поспіль" (Flapping detection).

### Г. Інтерфейс та аналітика
*   Dashboard зі статусом усіх сервісів (Зелений/Червоний).
*   Графік часу відповіді (Response Time) за останні 24 години.
*   Розрахунок SLA (Service Level Agreement) — відсоток доступності (наприклад, 99.9%).

---

## 4. Архітектура проєкту (Структура коду)

Це найважливіша частина для диплома. Використовуй **Clean Architecture**:

1.  **Domain Layer (Entities):**
    *   Structs: `Monitor`, `CheckResult`, `User`, `Alert`.
    *   Interfaces: `Checker` (з методом `Check()`), `Repository`, `Notifier`.
2.  **Service Layer (Business Logic):**
    *   Логіка запуску перевірок.
    *   Аналіз результатів (Up/Down).
    *   Логіка повторних спроб (Retries).
3.  **Repository Layer (Database):**
    *   SQL-запити до SQLite.
4.  **Delivery Layer (Transport):**
    *   HTTP Handlers (REST API).
    *   SSH Client implementation.

**Приклад інтерфейсу (ключовий момент):**
```go
type Checker interface {
    // Check виконує перевірку і повертає результат (успіх/помилка, затримка, повідомлення)
    Check(ctx context.Context) (bool, time.Duration, string, error)
    // Type повертає тип перевірки (HTTP, SSH, PING)
    Type() string
}
```
Це дозволить тобі легко додавати нові типи перевірок, не змінюючи ядро програми (принцип Open-Closed з SOLID).

### Детальний графік (Roadmap для новачка)

#### Етап 1: Вивчення основ Go (2-3 тижні)
Тобі не треба вчити *всю* мову. Для твого проєкту треба зрозуміти конкретні речі:

    *   *Ресурс:* Пройди [A Tour of Go](https://go.dev/tour/) (це офіційний інтерактивний туторіал, займе 1-2 вечори).

#### Етап 2: MVP — "Гидке каченя" (2 тижні)
Пишеш код, який працює, але виглядає страшно.
1.  **Тиждень 3:** Написати програму, яка читає список сайтів (хардкод у масиві) і пінгує їх циклом `for`.

### Де ти застрягнеш (Ризики)

Оскільки ти новачок, ось підводні камені, які з'їдять найбільше часу:

1.  **Конкурентність (Concurrency):** Ти напишеш код, який запускає 100 горутин, і програма впаде з помилкою `panic: runtime error`. Це нормально. Тобі доведеться розібратися з `sync.Mutex` (блокуванням ресурсів), щоб дві горутини не писали в одну змінну одночасно.
2.  **Структура проєкту:** У Go немає жорсткого стандарту, як у Django/Laravel. Ти можеш заплутатися, де створювати файли. *Порада:* Почни з пласкої структури (все в `main.go`), а розноси по папках тільки коли файл стане більше 500 рядків.
3.  **Контексти (`context`):** Тобі треба буде скасовувати завислі запити. Якщо сервер не відповідає, горутина має "померти", а не висіти вічно. Це концепція `Context` у Go, її треба зрозуміти.

### Чому Go — це найкращий вибір навіть для новачка?
*   **Він не дає тобі вистрелити собі в ногу.** Компілятор Go дуже суворий. Якщо ти створив змінну і не використав її — помилка. Якщо імпортував бібліотеку і забув про неї — помилка. Це бісить спочатку, але це економить години дебагу потім.
*   **AI (Copilot/ChatGPT) пише Go ідеально.** Go — дуже шаблонна мова. Ти можеш попросити AI: *"Write a Golang function to connect via SSH and execute command using crypto/ssh"* — і він видасть робочий код, який треба просто скопіювати.

### Порада для старту
Не намагайся писати "правильно" з першого дня. Пиши **"гівнокод"**, головне — щоб працювало. В університеті важливий **результат** (робоча програма), а не ідеальна чистота коду.
